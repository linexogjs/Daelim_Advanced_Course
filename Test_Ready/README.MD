## 1. 브로커 아키텍처의 장점과 단점을 기술하시오.

### ✅ 장점
- **이질적인 컴포넌트 연결 가능**: 다양한 언어와 플랫폼을 사용하는 컴포넌트 간 통신 지원.
- **모듈화 용이**: 컴포넌트 간 결합도를 낮추어 유지보수와 확장성 증가.
- **재사용성**: 서비스를 브로커를 통해 여러 클라이언트가 공유 가능.

### ❌ 단점
- **복잡성 증가**: 시스템 구조에 브로커가 추가되어 구조가 복잡해질 수 있음.
- **성능 저하 가능성**: 브로커를 거치는 추가 단계로 인해 지연이나 병목 현상 발생 가능.
- **디버깅 어려움**: 문제 발생 시 원인 추적이 어렵고 복잡할 수 있음.

## 2. 상용 소프트웨어를 개발하고자 할 때, 본인이 생각할 때 활용이 적합한 순서대로 GPL, LGPL, Apache 오픈소스 라이선스를 나열하고 그 이유를 기술하시오.

### 🔢 적합한 순서
**Apache → LGPL → GPL**

### 💬 이유
- **Apache License**
  - 수정/배포 시 소스 공개 의무 없음.
  - 상업용 소프트웨어에 적합하며 특허 보호 제공.
- **LGPL (Lesser GPL)**
  - 라이브러리에만 공개 의무가 있어 일부 상용 소프트웨어에 사용 가능.
- **GPL**
  - 코드 수정/링크 시 전체 소스코드를 공개해야 하므로 상업적 사용에 제약이 큼.

## 3. 스크럼 방법론에서 제품 책임자, 스크럼 마스터, 스크럼 팀의 역할에 대해 기술하시오.

### 제품 책임자 (Product Owner)
- 고객의 요구를 수집하고 정리하여 제품 백로그 관리.
- 개발 우선순위 설정 및 제품 가치 극대화 담당.

###  스크럼 마스터 (Scrum Master)
- 스크럼 실행을 촉진하고 팀의 장애물을 제거.
- 스크럼 규칙 준수 및 팀이 스스로 조직될 수 있도록 지원.

### 스크럼 팀 (Scrum Team)
- 실제 제품을 개발하는 개발자 중심의 팀.
- 스프린트 목표 달성을 위해 자율적으로 작업 수행.

---

## 4. CMM의 5단계에 대해 기술하시오.

1. **초기 (Initial)**  
   - 프로세스가 정의되지 않고 예측 불가능. 개인 역량에 의존.

2. **관리 (Managed)**  
   - 프로젝트가 계획되고 일정 및 자원 통제가 이루어짐.

3. **정의 (Defined)**  
   - 조직 수준에서 표준화된 개발 프로세스를 문서화하고 따름.

4. **정량적 관리 (Quantitatively Managed)**  
   - 정량적 데이터로 품질 및 성능 측정. 예측 가능성 확보.

5. **최적화 (Optimizing)**  
   - 지속적 개선을 위한 피드백과 혁신이 이루어짐.

---

## 5. 소프트웨어 개발이 어려운 이유는 무엇이라고 생각하는가?

- **요구사항 변경이 빈번함**: 고객의 요구가 자주 바뀜.
- **복잡하고 추상적임**: 실체가 없는 논리적 시스템이기 때문에 전체 구조 파악이 어려움.
- **커뮤니케이션 오류**: 사용자와 개발자 간 오해 발생 가능성.
- **완벽한 테스트의 어려움**: 모든 상황을 예측하고 테스트하기 힘듦.
- **기술 변화 속도**: 기술 발전이 빨라 빠르게 구식이 될 수 있음.



## 수요일2 창의융합프로젝트

### 월드와이드 웹
    - 월드와이드 웹 (World Wide Web)
    - 인터넷에 연결된 컴퓨터들이 하이퍼텍스트 형시긍로 편현된 다양한 정보를 효과적으로 이용할 수 있도록 구성한 전 세계적인 시스템
    - 세계 최고의 거미집 또는 거미집 모양의 망이라는 뜻
### 웹의 탄생 
    - 89년, 유럽입자물리연구소의 팀 버너스 리가 과학자들 사이에 쉽게 정보를 주고받기 위한 목적으로 ㅈ어보 관리 제안서를 발표 (최초의 인터넷 기반 하이퍼 텍스트 프로젝트 제안)
### 초창기 웹
    - 단순한 텍스트와 링크위주
    - 하이퍼텍스트 : 문서 중간중간에 특정 키워드를 두고 문자나 그림을 상호 유기적으로 결합하여 연결시킴으로써, 서로 다른 문서라 할지라도 하나의 문서인 것처럼 보이면서 참조하기 쉽도록 하는 방식
    - 하이퍼링크 : 한 페이지에서 다른 페이지의 문서로 쉽게 이동
    - 웹 서핑, 웹 브라우징:  하이퍼링크를 따라 이동하는 것
### HTTP
    - hypertext transfer protocol
    - 인터넷에서 하이퍼텍스트 문서를 교환하기 위하여 사용되는 통신규약
    - 인터네셍서 웹 서버와 사용자의 인터넷 브라우저 사이에 문서를 전송하기 위해 사용되는 통신 규약
    - 팀 버너스 리가 웹을 만들면서 개발한 프로토콜
    - 인터넷 주소를 지정할 때 www. 와 같이 하는 것은 www로 시작되는 인터넷 줒소에서 하이퍼텍스트 문서의 교환을 http 통신규약으로 처리하라는 의미
### HTTP 0.9
    - 0.9 버전의 http 서버에서는 단순히 읽기 기능만 지원
    - 인터넷을 통하여 가공되지 않은 데이터를 전송하기 위한 단순한 프로토콜
### http1.0과 1.1
    - 데이터에 대한 전송과 요구 응답에 대한 수정 등 가공된 정보를 포함하는 프로토콜로 개선
    - 현재 웹에서 주로 사용
    - 1.0 : 96년 5월에 완성, 메소드는 get, head, post 방식만 지원
    - 1.1 : 2001년 공식 발표돠어, 메소드는 option,get, head, post, delete, trace, connect 방식 지원

### OWASP
    - Open Web Application Securiy Project : 웹 애플리케이션 보안에 대한 정보를 공유하고 체계를 세우는 자발적인 온라인 공유 사이트
    - 2001년 12월, 마크 커페이와 데니스 그로브스 등에 의해 탄생
    - 우리나라는 2011년 1월 owasp korea chapter 이사회가 조직되어 활동 전개

### SQL 
    - Structured Query Language
    - SQL문, 쿼리문, SQL명령으로 불림
    - 74년에 IBM의 도널드 챔벌린과 레이먼드 보이스가 개발
    - 관계형 데이베이스 관리 시스템(RDBMS)의 데이터를 조회, 수정 및 관리하기 위해 만들어진 데이터베이스 표준 언어
    - 대다수 상용 또는 공개 데이터베이스 관련 프로그램은 SQL을 표준으로 함
    - 데이터베이스에 직접 접속하여 명령을 내리거나 PHP와 같은 프로그램에 삽입하여 사용

### SQL 인젝션 공격
    - SQL 인젝션 공격
    - 웹 애플리케이션과 데이터베이스가 연동되는 부분에 공격자가 임의의 SQL 명령어를 삽입하여 공격
    - 클라이언트가 응용 프로그램에 SQL 쿼리르 ㄹ통하여 악의적인 삽입하는 것
    - 예시 : 사용자 로그인 부분, 게시물 검색 부분, 우편번호 검색 부분, 자료실 등
### SQL 인젝션 공격자의 목적
    - 신원 위장 / 데이터베이스 서버의 관리자로 위장
    - 기존 데이터 변조
    - 잔액 변경과 같은 부인 문제 발생
    - 시스템 사으이 모든 데이터의 완전한 공개
    - 데이터의 삭제 또는 데이터의 무효화
### SQL 인젝션 공격의 역사
    - 98냔 12월, 온라인 잡이 ... rfp 팀이 웹 애플리케이션괃 에이터베이스 간의 취약점과  SQL문을 변조할 수 있다는 가능성을 언급
    - rfp가 'SQL Injection '이라는 문서 공개
    - 02년 3월, SQL 인젝션 공격으로 약 20만명의 신용카드 사용자 정보 유출
    ....

### 악성코드
    - 악성코드 (Malware, Malicious Software) : 악성 프로그램
    - 악의적인 목적으로 제작된 컴퓨터에서 작동하는 모든 형태의 코드
    - 의도적으로 사용자에게 피해를 주기 위해 만들어진 프로그램, 매크로, 스크립트.
### 동작에 의해 악성 코드 분류
    - 바이러스 
    - 웜
    - 트로이목마
### 바이러스
    - 사용자의 컴퓨터에서 프로드램이나 실행 가능한 부분을 몰해 수정하여 자신 또는 자신의 변형을 복사
    - 악성코드의 가장 기본적인 형태
### 1세대 원시형 바이러스
    - 처음 컴퓨터 바이러스가 등장한 시점의 가장 원시적인 형태
    - 자기 복제와 데이터 파괴의 단순한 기능
    1. 부트 바이러스
        - 플로피디스크/하드디스크의 부트 섹터에 감염
        - 부팅 시 자동으로 동작, 부팅 후 사용되는 모든 프로그램을 감염
    2. 파일 바이러스
        - 일반적으로 실행 파일 또는 오버레이 파일, 디바이스 드라이버 등에 감염
        - 하드디스크 부팅이 일반화되면서 전체 바이러스의 80% 이상을 차지
### 2세대 암호형 바이러스
    - 바이러스 코드를 쉽게 파악하여 제거하지 않도록 암호화한 바이러스
    - 1세대는 특정 패턴이 존재하여 백신은 이 패턴을 통해 바이러스 진단 후 삭제(치료) -> 2세대에서는 코드 암호화를 통해 진단할 수 없도록 함 -> 백신을 바이러스 실행 시 복호화된 코드가 메모리에 올라오는 점을 이용하여 진단 치료
### 3세대 은폐형 바이러스
    - 바이러스에 감염된 파일이 일정 기간 잠복기를 가지도록 만든 것
    - 바이러스가 확산되기도 전에 활동하기 시작하면 다른 시스템으로 전파되기 힘들기 때문
### 4세대 다형성 바이러스
    - 백신 프로그램은 바이러스 파일 안의 특정한 식별자로 바이러스 감염 여부를 판단
    - 조합 프로그램을 암호형 바이러스에 추가하여 프로그램이 실행될 때마다 바이러스 코드를 ㅂ녀경하고 식별자를 구분하기 어렵게 함
    - 제작 및 진단이 어려움
### 5세대 매크 바이러스
    - MS 오피스 프로그램의 매크로 기능으로 감염되는 바이러스
    - 엑셀, 워드 등의 문서파일을 열 때 감염
    - 비주얼 베이지 스크립트 (VBS)로 많이 제작

### 매크로 바이러스 증상
    - 문서가 정상적ㅇ로 열리지 않거나 암호 설정
    - 문서 내용에 깨진 글자나 이상한 문구 포함
    - 매크로 매뉴가 실행할 수 없게 잠겨 있음
    - 액셀, 워드 작업 중 VB 편집기의 디버그 모드 실행

### 차세대 바이러스
    - 스크립트 형태의 바이러스가 더욱 활성화되어 네트워크와 메일을 이용하여 전파
    - 단순한 데이터 파괴 및 감염 기능을 넘어서 사용자 정보를 빼내거나 백도어 기능을 가진 웜을 형태로 진화
### 웜
    - 인터넷 또는 네트워크를 통해 컴퓨터에서 컴퓨터로 전파
    - 컴퓨터의 취약점을 이용하여...스스로 증식하여 전파
    - 확산 방법
        - 이메일 첨부 파일 형태
        - 운영체제나 프로그램의 보안상 취약점을 이용
        - 채팅 프로그램, p2p 파일 공유 프로그램, 이메일 스크립트 기능, 네트워크 공유 기능 등의 취약점을 이용
### 전파 형태에 따른 웜 유형
    - 메스메일러형 웜
        - 자기 자신을 포함하는 대량 메일을 발송하여 확산
        - 제목없는 메일이나 특정 제목의 메일을 전송하고 사용자가 이를 읽었을 때 감염
    - 시스템 공격형 웜
        - 운영체제 고유의 취약점 이용
        - 시스템 내부 정보 파괴 기능, 컴퓨터를 사용 불가능한 상태로 만듦, 외부 공격자 접속을 위한 악성 코드 설치 등..
    - 네트워크 공격형 웜
        - 특정 네트워크나 시스템에 대해 syn flooding, smurf와 같은 dos 공격 수행
        - ddos 공격을 위한 봇 형태로 발전
        - 증상 :  네트워크가 마비되거나 급격히 느려짐 /  넽,워크 장비가 비정상적으로 동작
### 트로이 목마
    - 악성 루틴이 숨어있는 프로그램 : 겉보기에는 정상적인 것 같지만 사용자가 실행하면 악성코드 실행
    - 일종의 사회공학 기법
    - 스파이웨어, 랜섬웨어 등 어떤 악성 코드도 포함될 수 있지만 주로 백도어로 사용

### 스테가노그래피

    - 미리 정학 약속에 따라 특정 데이터를 원래의 것과 전혀 관련이 없는 데이터로 해석

### 워터마크
    - 페이지 전면에 옅은 색읗로 무늬를 나타내는 것
    - 예시 : 회사 문서에 표시되는 회사 로고, 인터넷에서 발급한 주민등록본에 표시된 무궁화 문양
    - 영상이나 오디오 파일에도 삽입 가능
    - 사용자가 알아볼 수 있도록 표시 / 저작물이 조작되지 않도록 인지할 수 없는 방식으로 표시

### 네트워크 보안

### DOS
    - Denial of Service
    - 서버의 성능을 크게 떨어뜨리거나 서버를 정지시키는 방법을 통해 서버의 정상적인 작동을 방해하는 공격
    - 공격 대상이 수용할 수 있는 능력 이상의 정보를 제공하거나 사용자 또는 네트워크 용량을 초과 유도
### DOS 공격기법
    - Ping of Death
    - Syn Flooding Attacks
    - Smurf Attacks
    - Land Attacks
    - Teardrop Attacks
### Ping of Death
    - 윈도우 95, 98 레드헷 리눅스 6.0 이하 버전 대상
    - 초기 DOS 공격
### Ping of Death 공격과정
    1. Ping을 이용하여 ICMP 패킷의 크기를 정상보다 아주 크게 만듦
    2. 패킷은 네트워크를 통해 라우팅되어 공격 네트워크에 도달하는 동악 아주 작은 조각으로 쪼개짐
    3. 공격 대상은 조직화된 패킷을 모두 처리해야 하므로 정상적인 ping보다 부하가 훨씬 많이 걸림

### Ping of Death 공격원리
    - 라우팅은 패킷을 전달할 때 특성이 똑같은 네트워크를 지나지 않음
    - 네트워크마다 최대로 전송이 가능한 패킷의 길이 달라 최대 전송 가능한 패킷의 길이가 작은 네트워크를 리잔면 데이터는 더 작게 분할 됨
    - 한번 분할된 패킷은 다시 커지지 않음 <- 패킷을 저장한 후 다음에 들어온 패킷 데이터를 재조합해야 하는데 라우터 성능에 치명적일 정도 높은 부하를 야기함

### TCP/IP 4계층과 프로토콜
    - 응용계층 : DHCP, DNS, FTP, SMTP, SNMP, Telnet
    - 전송계층 : TCP, UDP
    - 인터넷계층 : IP, ICMPM IGMP, ARP, RARP
    - 네트워크 인터페이스 계층 : L2TP..
### ICMP    
    - 인터넷 계층 프로토콜
    - 호스트 서버와 인터넷 게이트웨이 간 메시지 제어와 오류에 관한 정보를 전송하기 위한 프로토콜
    - 활용 예 : PING

### SYN Flooding Attacks 대책
    - 시스템 패치 적용
    - 칩임 탐치 시스템(IDS) 또는 침임 차단 시스템(IPS) 설치
    - 짧은 시간 내 동일한 형태의 패킷 전송 -> 해당 IP 주소 차단 조치
    - Syn_Cookie 설정 - 암호화 기술을 이용하여 인증 정보가 포함된 시퀀스 넘버 생성 및 전송
### Teardrop Attacks 
    - 이 프로토콜을 수신한 패킷 순서나 손실 패킷 유무를 체크하여 재전송 요구와 수정을 반복함
    - 이 공격은 재전송 요구와 수정 반복을 계속하게 함으로써 시스템 자원을 고갈시킴
    - 유사공격 - Boink(보잉크),  Bonk(봉크)
### 패킷 시퀀스 넘버와 배치
    - 시퀀스 넘버를 바꾸고 중첩과 빈 공간을 만들어 시퀀스 넘버가 복잡해지도록 섞음
    - 공격 대상이 패킷화된 데이터를 재조합하는데 혼란이 생겨 CPU에 과부하가 걸리게 됨
### DDOS 
    - 여러 대의 컴퓨터를 일제히 동작시켜 특정 사이트/시스템을 공격
    - 엄청난 분량의 패킷을 동시에 범람시킴으로써 네트워크 성능 저하나 시스템 마비를 가져오게 됨

### DDOS 특성
    - 피해 양상이 심각하나 확실한 대책이 없음
    - 공격자의 위치와 구체적인 발원지를 파악하는 것은 매우 어려움
    - 대부분의 DDOS 공격은 자동화된 툴을 이용
### DDOS 공격 구성
    - 공격자 : 공격을 주도하는 해커의 컴퓨터
    - 마스터 
        - 공격자에게 직접 명령을 받는 시스템, 여러 대의 에이전트 관리
        - 핸들러 시스템 : 마스터 시스템 역할을 수행하는 프로그램
    - 에이전트
        - 공격 대상에 직접 공격을 가하는 시스템
        - 데몬 프로그램  : 에이전트 시스템 역할을 수행하는 프로그램
### 공격 대응 방법
    - 방화벽 설치와 운영
    - IPS의 설치와 운영
    - 안정적인 넽,워크 설계
    - 시스템 패치
    - 스캐닝
    - 서비스별 대역폭 제한

### 방화벽
    - 방화벽, 침임차단시스템
    - 관리자는 방화벽에 통과시킬 접근과 그렇지 않은 접근을 명시
    - 룰셋을 통해서 수행
    - 룰셋은 방화벽을 기준으로 보호하고자 하는 네트워크 외부와 내부에 존재하는 시스템들의 IP와 포트단위로 이루어짐

### 해킹 DDoS에 안전한 방화벽 룰셋
    - 최소한 서비스만 제공하고 사용하지 않는 서비스의 포트는 닫음
    - 외부 네트워크에서 들어오는 패킷을 출발지 주소가 내부 네트워크에 존재하는 주소지와 일치할 경우 차단
    - 구성 요소 간 트러스트를 금지 - 인증 없이 시스템에 접속할 수 있은 내부/외부 사용자를 허용하지 않음

### DoS/ DDoS 대응 - 안정적인 네트워크 설계
    - 방화벽 차단과 IDS, IPS 탐지가 실패한 경우
    - 내부 시스템은 공격에 어느 정도 저항성이 있어야 함
    - CAR - 라우터로 일정량 이상의 패킷이 들어옳 경우 그 이상의 패킷을 통과시키지 않음
    - 각 장비 간 로드밸런싱 설정
    - 백업 서버 운영
### 시스템 패치
    - 시스템에서 바이러스와 해킹 공격에 취약한 점이 발견되면 각 업체에서 패치를 발표
    - 해당 패치가 어떤 작용을 하는지 이해한 뒤 설치해야 함 (백업 필수)
### 스캐닝
    - DDoS 공격, 백도어 등에 의하여 은밀하게 설치된 디렉토리, 루트킷 커널 모듈 등을 감시하기 위한 시스템 분석
### 서비스별 대역폭 제한
    - 각 서비스별 대역폭을 조절하여 특정 서비스에 대한 공격이 이루어지더라도 나머지 서비스에 대한 영향을 최소화 할 수 있음
    - 공격이 이루어지고 있는 서비스도 공격 성공에 필요한 최소 대역폭을 얻을 수 없어 공격에 성공하기 어려움
### 스나핑
    - 수동적 공격
    - 모든 종류의 도청 /  엿듣기가 해당
        - 전화선이나 UTP에 탭핑해서 전기 신호를 분석하여 정보를 찾아냄
        - 전기신호을 템페스트 장비를 이용해 분석하는 일
    - ARP Redirect 공격 / ICMP Redirect 공격/ 스위치 재밍
### 프러미스큐어스 모드
    - 프러미스큐어스 모드
    - MAC 주소와 IP 주소에 관계없이 모든 패킷을 버리지 않고 읽는 것 -> 스니핑이 가능한 모드
    - 리눅스/유닉스 - 랜 카드에 대한 모든 설정이 가능
    - 윈도우 - 프러미스큐어모드가 없으며 스나핑을 위한 드라이버를 따로 설치해야 함
### ARP Redirect 공격
    - 공격자가 자신을 라우터라고 속이는 것
    - 공격자는 위조된 패킷을 이용하여 자신의 MAC 주소가 라우터라고 주기적으로 브로트캐스트 함
    - 공격자 자신은 원래 라우터의 MAC 주소를 알고 있어야 함
    - 공격자는 받은 모든 패킷을 다시 라우터로 릴레이 해줌
    - 기본적으로 인터넷 계층 공격으로, 랜에서의 공격
### ARP 
    - ARP
    - 논리적인 주소인 IP주소를 물리적인 주소인 MAC 주소로 변환시켜주는 프로토콜
    - 최종적으로 수신지 네트워크 카드로 신호가 전송되기 위해서는 네트워크 카드의 물리 주소의 MAC 주소를 알아야 함

### 스위치 재밍
    - 스위치를 직접 공격
    - MAC 테이블을 위한 캐시 공간에 버퍼 오버플로우 공격을 실시
    - 잃부 고가의 스위치는 MAC 테이블의 캐시와 연산 장치가 사용하는 캐시가 독립적으로 나뉘어 있어 스위치 재밍 공격이 통하지 않음
### 스니퍼 탐지
    - 스니핑이 프러미스큐어스 모드에서 작동함에 착안
    -  Ping을 이용한 탐지
    - ARP를 이용한 탐지
    - DNS를 이용한 탐지
    - 유인을 이용한 탐지
    - ARP watch 를 이용한 탐지
    
### 수동적인 대응책 - 암호화
    - 암호화 방법은 서비스에 따라 다름
    - SSL 
        - 암호환된 웹 서핑 지원
        - 많은 웹 서버 브라우저에 구현되어 있음
        - 40비트, 128비트,256비트 암호화 사용
        - 현재 우리나라 금융 거래 사이트의 대부분은 128비트 암호화 방법을 사용
    - PGP, PEM, S/MINE
        - 이메일을 전송할 때 사용하는 암호화 방법
        - PGP : IDEA와 RSA 사용, 서로에 대해 신뢰를 가지고 서로가 서로에 대한 인증을 해주는 제3자가 되는 구조 (Web Of Trust)
        - PEM : 공개키 암호화 표준과 공인된 인증기관을 통하여 암호화
        - S/MINE : PKCS, X.509 이용, 이메일 표준인 MINE 형식에 암호화 기능
    - SSH  
        - 텔넷과 같은 서비스 암호화를 위해 사용하는 것
        - OpenSSL 라이브러리가 SSH를 지원
        - 텔넷보다는 훨씬 더 안전
    - VPN
        - 원래는 한 회선을 여러 회사가 공유하여 비용을 절감하려는 목적으로 개발
        - 각 회사의 사내 정보 보호를 위하여 암호화된 트래픽 제공
        - VPN을 제공하는 시스템이 해킹을 당할 경우 암호화되기 이전에 데이터가 스나핑이 될 수 있음

    - 속이다는 의미
    - 인터네싱나 로컬에서 존재하는 모든 연결에 스푸핑 가능
    - 정보를 얻어내기 위한 중간 단계의 기술
    - 시스템을 마비시키는 데 사용할 수도 있음
    - ARP 스푸핑/ IP 스푸핑/ DNS 스푸핑

### ARP 스푸핑
    - MAC 주소를 속이는 것
    - 인터넷 계층에서 작동해 공격 대상이 같은 랜에 있어야 함
### ARP 스푸핑 공격 대비책
    1. 시스템의 MAC 주소를 확인하여 테이블로 만들어 둠
    2. 브로드캐스트 ping을 네트워크에 뿌려 그에 답하는 모든 시스템을 캐시에 기록
    3. apa-a로 현재 IP주소 값과 MAC 주소의 대칭값을 비교하여 엉뚱한 MAC 주소로 맵핑되어 있는 항목을 확인
### ARP 스푸핑 공격 대비책 - static MAC 주소 설정
    - ARP 스푸핑 공격이 들어오도 변하지 않는 IP와 MAC 주소 설정 -> 관리가 어려운 단점
### IP 스푸핑
    - IP 주소를 속이는 것
    - 트러스트 (trust) - 시스템에 접속할 땝 자신의 IP 주소로 인증하면 로그인 없이 접속이 가능하게 만든 것 -> IP만 일치한면 인증 우회가 가능

### IP 스푸핑 보안 대책
    - 가장 좋은 보안 대책은 트러스트를 사용하지 않는 것
    - 트러스트를 사용해야 한다면 트러스트된 시스템의 MAC 주소를 Static으로 지정
    - 최근 계정의 패스워드가 같을 때만 패스워드를 묻지 않도록 변경
    - SSH를 사용하도록 권고하기 때문에 IP 스푸핑 공격이 이루어지지 않음
    
### DNS 스푸핑
    - 웹 스푸핑과 비슷
### DNS 
    - 응용 계층 프로토콜
    - 웹 사이트의 영문/한글로 된 도메인 네임과 숫자로 된 IP 주소 관계를 관리
### DNS 클라이언트와 서버
    - 학내 DNS 서버로부터 해당 정보를 얻지 못한다면 해당 정보를 등록하고 있는 다른 DNS 서버를 찾아서 그곳에 질의를 보냄
### 정상적인 DNS 서비스 과정
    1. 클라이언트는 접속하고자 하는 IP 주소를 문의하고자 DNS Query
    2. DNS 서버가 해당 도메인 이름에 대한 IP 주소를 클라이언트에 전송
    3. 클라이언트는 DNS 서버에서 받은 IP 주소를 바탕으로 웹 서버 접속

### DNS 스푸핑 보안 대책
    - host 파일 사용하기
        - 사이트에 접속하면 캐시에서 읽어들인 후 hosts 파일을 통해 도메인 이름에 대한 ip주소를 해석
        - hosts 파일에 중요한 사이트의  ip 주소를 확인하여 적어두면 dns 스푸핑 공격을 당하지 않음
    - DNS 서버에 대한 DNS 스푸핑 공격 방지
        - BIND를 최신 버전으로 바꿔서 해결

### 세션 하이재킹
    - 세션 하이재킹
    - 세션 가로채기라는 의미
    - 적극적인 공격
### TCP 세션 하이재킹 공격과정
    1. 클라이언트, 서버 모두 established 상태, 공격자는 sniffing으로 시퀀스 넘버 휙득
    2. 공격자 RST 패킷을 서버로 전송 -> 클라이언트는 established 유지, 서버는 잠시 closed
    3,4. 공격자와 서버는 새로운 시퀀스 넘버 교환
    5. 공격자, 서버 establisshed 상태, 클라이언트도 established 상태 유지
    6. 공격자는 기존의 시퀀스 넘버로 클라이언트와, 새로 생성한 스퀀스 넘버로 서버와 통신
### 애크 스톰
    - 클라이언트가 정상적인 패킷을 보내면 서버는 정상적이지 않은 시퀀스 넘버로 인식
    - 서버는 시퀀스 넘버를 맞추기 위해 ack 패킷에 server_my_seq와 server_client_seq를 담아 보냄
    - 클라이언트는 서버가 보내온 server_client_seq가 자신의 client_my_seq와 다름을 확인
    - 클라이언트는 서버에 client_my_seq와 client_server_seq가 담긴 ack를 전송
    - 이러한 과정이 무한 반복 -> 공격자는 잘못된 패킷이 전달되지 않도록 arp 스푸핑을 해두고 공격을 실시
